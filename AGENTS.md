# AGENTS.md

このリポジトリで作業する AI エージェント向けの指示です。

## プロジェクト概要

- 目的は、シンプルなマルチユーザー向け Markdown オンラインエディタの試作品を作ることです。
- 現時点の推奨構成は `Vite + React`、`CodeMirror 6`、`react-markdown`、`Supabase Auth`、`Supabase PostgreSQL`、`Cloudflare Pages` です。
- 初期試作品では、本格的なリアルタイム共同編集は対象外です。保存時の競合検知を優先してください。

## 実装方針

- まずは小さく動く試作品を優先し、過度な抽象化や複雑な設計を避けてください。
- ドキュメント管理は最小構成にしてください。フォルダーやタグ機能は追加しないでください。
- Markdown 編集体験を主目的とし、エディタは原則として CodeMirror 6 を優先してください。
- 複数ユーザー対応は Supabase Auth と DB の権限管理を前提にしてください。
- 権限管理では Supabase Row Level Security の利用を前提にしてください。

## データモデルの前提

最小構成は次を基準にしてください。

```text
users
- id

documents
- id
- owner_id
- title
- body_markdown
- created_at
- updated_at

document_members
- document_id
- user_id
- role  // owner, editor, viewer
```

## 競合制御

- 初期実装では、`updated_at` を使った楽観的競合チェックを採用してください。
- 他ユーザーが先に更新していた場合は、自動マージせず、再読み込みを促す UI にしてください。
- Yjs などを使った本格的な同時編集は、明示的に依頼されるまで導入しないでください。

## コーディング規約

- 変更は最小限にし、依頼内容と直接関係しないリファクタリングは避けてください。
- 既存のスタイル、命名、ディレクトリ構成に合わせてください。
- セキュリティ上重要な処理、特に認証・認可・DB アクセスはクライアント側だけに依存しないでください。
- 環境変数や API キーなどの秘密情報をコミットしないでください。
- 必要に応じて `.env.example` を更新してください。

## 検証

- 実装後は、変更範囲に最も近いテスト、型チェック、lint、ビルドを優先して実行してください。
- テスト基盤がない場合は、新しいテスト基盤を勝手に導入せず、手動確認手順を簡潔に示してください。
- 失敗が既存の無関係な問題に起因する場合は、修正せずに内容を報告してください。

## ドキュメント

- 仕様やセットアップ手順を変更した場合は、`README.md` など関連ドキュメントも更新してください。
- 実装判断に影響する前提や制約は、コードだけでなくドキュメントにも残してください。
